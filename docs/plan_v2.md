# Dark Maze — Plan v2 (Authoritative Spec)

---

# 1. Goal

Build a client-server dungeon exploration system where:

* The **player explores blindfolded**
* The **server is authoritative**
* A **GM controls and observes the world**
* A **maze can be designed, edited, and played**

The system must support:

* deterministic turn-based resolution
* immersive sensory feedback (walls, sound, proximity)
* GM-driven or AI-driven monster behavior

---

# 2. Core System Model

## 2.1 World Structure

* Maze is a rectangular grid: `(width x height)`

* Each cell contains:

  * walls: `{n, e, s, w}` (boolean)
  * optional:

    * hazard
    * item
    * exit flag
    * entry flag

* Edges between cells must remain **consistent (shared walls)**

---

## 2.2 Actors

### Player

* position `(x, y)`
* facing (N/E/S/W)
* stats:

  * perception_bonus
  * stealth_bonus

### Monsters

* same movement model as player
* additional:

  * monsterType
  * actionPool
  * explored_cells
  * last_sound_hint

### Monster Types

Define:

* maze_proficiency (pathfinding intelligence)
* sound_homing (response to sound)
* sound phrases (for player feedback)

---

## 2.3 Actions

All actors share:

* turn_left
* turn_right
* forward
* wait
* vocalize

Rules:

* turn_left/right → no time progression (K = 0)
* forward → moves one cell, K depends on pace
* wait → no movement, K = 3, improves listening
* vocalize → no time progression (K = 0)

---

## 2.4 Time Model

* No global clock
* Time advances only on:

  * forward
  * wait

Each action produces **K micro-increments**

During each increment:

1. monsters gain action pool
2. monsters act (partial + new actions)
3. sound events propagate
4. hearing is evaluated

---

## 2.5 Hearing System (Simplified MVP)

* Noise events generated by:

  * forward (loudness depends on pace)
  * wait (minimal)
  * monster vocalizing (random)

* Hearing uses:

  * d20 perception vs stealth
  * propagation modifier

On success:

* player receives:

  * direction (8-way relative)
  * distance band (text)
  * phrase (monster sound)

---

## 2.6 Visibility Rules

Player receives ONLY:

* walls relative to facing:

  * front / left / right / behind
* current cell info
* heard cues

Player NEVER receives:

* map
* monster positions
* monster types (optional later)

# 3. Architecture

## 3.1 Server (Python)

* FastAPI + WebSocket
* authoritative state
* modules:

  * maze
  * resolver
  * acoustics
  * actors
  * protocol models

## 3.2 Client

### Player Client

* browser-based
* Apple IIe-style interface, Wizardry style
* keyboard-driven actions

### GM Client (Play Mode)

* map view
* monster control
* pause/resume
* debug info

### GM Design Client

* graphical maze editor
* generation controls
* placement tools

---

# 4. Design Mode (Authoring System)

## 4.1 Workflow

1. Define maze size
2. Initialize **all-walls grid**
3. (Optional) run maze generation algorithm
4. Manually edit walls via path-drag:

   * drag = open walls
   * shift+drag = close walls
5. Place:

   * player start
   * exit
   * monsters
   * items / hazards
6. Apply template to server

---

## 4.2 Generation System

* algorithm registry:

  * recursive_backtracker (MVP)

* parameters:

  * seed
  * future: density, bias

* regeneration replaces wall topology

  * warn GM first

---

## 4.3 Data Format

Template includes:

* width / height
* grid
* player_spawn
* exit
* monsters
* monster_types
* tuning parameters

---

## 4.4 Server API (Design)

WebSocket or REST:

* design.apply_template
* design.generate_maze
* design.state

---

# 5. Play Mode

## 5.1 Session States

* design
* play
* paused

## 5.2 Rules

* player actions rejected in design
* player actions rejected when paused
* GM controls pause/resume

---

## 5.3 Resolver

For each player commit:

* resolve player action
* run K increments
* update monster actions
* evaluate hearing
* check win/lose

---

# 6. Player Experience

## 6.1 Interface

* text-mode (Apple II Wizardry inspired)
* wall indicators:

  * top / bottom bars
  * side vertical text
* no graphics map

## 6.2 Input

* W / Up → forward
* A / Left → turn left
* D / Right → turn right
* Space → wait
* ! → vocalize

## 6.3 Feedback

* immediate input feedback (flash/sound)
* hearing cues shown per action

---

# 7. GM Experience (Play)

* live map
* player + monster positions
* monster control:

  * AI / manual
* hearing debug
* pause/resume

---

# 8. Execution Phases

## Phase 1 — Core Engine (MVP)

* maze data model
* player movement
* wall logic
* simple client view
* no monsters OR simple random monster

## Phase 2 — Turn System + Hearing

* K increments
* noise propagation
* hearing system
* basic monster AI

## Phase 3 — GM Play Tools

* map view
* pause/resume
* monster control

## Phase 4 — Design System

* maze editor UI
* generation algorithms
* template system

## Phase 5 — Advanced Features

* richer AI
* multiple monster types
* tuning system
* polish

---

# 9. Non-Goals (for now)

* multi-floor mazes
* advanced UI graphics
* undo/redo system
* multiplayer players
* perfect AI behavior

---

# 10. Key Principles

* server is authoritative
* player knowledge is strictly limited
* system is deterministic
* design mode and play mode are separate
* plans must execute in phases, not all at once
